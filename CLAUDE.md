# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Instagram-like app ("NeNe") built as a pnpm/Turborepo monorepo:

- `apps/backend` — NestJS API (auth, posts, uploads), Drizzle ORM over Postgres, exposes a tRPC router.
- `apps/web` — Next.js 16 (App Router) frontend, shadcn/radix-ui components, Tailwind v4.
- `packages/trpc` — shared package holding the generated tRPC `AppRouter` type (`src/server/server.ts`), consumed for end-to-end type safety between backend and frontend. **This file is auto-generated by `nestjs-trpc`** (see `TRPCModule.forRoot({ autoSchemaFile: ... })` in `app.module.ts`) — don't hand-edit it, change the `@Router`/`@Query`/`@Mutation` decorators in the backend instead and let it regenerate.
- `packages/eslint-config`, `packages/typescript-config` — shared lint/tsconfig presets consumed via `workspace:*`.

## Commands

Run from repo root (uses Turborepo to fan out per package):

```sh
pnpm dev            # turbo run dev (all apps, persistent/watch)
pnpm build          # turbo run build
pnpm lint           # turbo run lint
pnpm check-types    # turbo run check-types
pnpm format         # prettier --write "**/*.{ts,tsx,md}"
```

Scope to one app with `--filter`, e.g. `pnpm exec turbo dev --filter=web` or `pnpm exec turbo dev --filter=backend`.

### Backend (`apps/backend`)

```sh
pnpm --filter backend dev          # nest start --watch
pnpm --filter backend build        # nest build
pnpm --filter backend test         # jest unit tests
pnpm --filter backend test:watch
pnpm --filter backend test:e2e     # jest --config test/jest-e2e.json
pnpm --filter backend lint         # eslint --fix
```

Run a single test file/case with jest directly, e.g. `cd apps/backend && pnpm test -- posts.service.spec.ts` or `pnpm test -- -t "test name"`.

Drizzle migrations (schema files are globbed from `src/**/schema.ts`):

```sh
cd apps/backend
pnpm drizzle-kit generate   # create a new migration from schema changes
pnpm drizzle-kit push       # push schema directly (dev)
```

Requires `DATABASE_URL`, `BETTER_AUTH_URL`, `BETTER_AUTH_SECRET`, `UI_URL` in `apps/backend/.env`.

### Web (`apps/web`)

```sh
pnpm --filter web dev           # next dev --port 3000
pnpm --filter web build
pnpm --filter web lint          # eslint --max-warnings 0
pnpm --filter web check-types   # next typegen && tsc --noEmit
```

Requires `API_URL` in `apps/web/.env`.

## Architecture

### Backend module layout

Each feature is a self-contained Nest module under `src/`:

- `auth/` — `better-auth` wired in via `@mguay/nestjs-better-auth`'s `AuthModule.forRootAsync` in `app.module.ts`, using the Drizzle adapter (`drizzleAdapter`) against the shared DB connection. `auth/schema.ts` defines the drizzle tables `better-auth` needs (`user`, `session`, `account`, `verification`) plus relations to app tables (e.g. `posts`). A global `APP_GUARD` (`AuthGuard` from `@mguay/nestjs-better-auth`) protects all routes by default.
- `auth/users/` — `UsersService`/`UsersModule` for looking up app users (distinct from the auth tables above); imported by other feature modules (e.g. `posts`) that need user data.
- `posts/` — `posts.router.ts` is a **tRPC router** (`@Router()` class using `nestjs-trpc` decorators `@Query`/`@Mutation`/`@Input`), not a REST controller. Input/output validation and inferred types live in `posts/schemas/trpc.schema.ts` (Zod schemas). The Drizzle table definition lives separately in `posts/schemas/schema.ts`. Keep these two schema files distinct: `schema.ts` = DB table (drizzle), `trpc.schema.ts` = wire-format Zod schema for the router.
- `upload/` — file uploads via `@nestjs/platform-express`'s `MulterModule`, configured with disk storage in `upload.config.ts` (files land in `./uploads/images` with uuid-suffixed names).
- `database/` — single shared Postgres connection (`DATABASE_CONNECTION` token, provided via `DatabaseModule`) built with `drizzle(pool, { schema })`, where `schema` is the merge of every feature's schema module (`database.module.ts` re-exports this merged `schema` object — import it from there rather than re-merging).

### tRPC wiring

`TRPCModule.forRoot({ autoSchemaFile: '../../packages/trpc/src/server' })` in `app.module.ts` makes `nestjs-trpc` introspect all `@Router()` classes on boot and regenerate `packages/trpc/src/server/server.ts`, which exports the `AppRouter` type. That type is meant to be imported by the frontend for a type-safe tRPC client — as of now the web app has not wired up a tRPC client yet (`handleCreatePost` in `apps/web/app/page.tsx` is a stub), so a new tRPC-based data flow needs both the router change on the backend and the client-side hookup on the frontend.

### Frontend structure

- App Router pages live in `app/`; `middleware.ts` gates every route except `/login` and `/signup` behind a `better-auth` session cookie check (`getSessionCookie`).
- `lib/auth/client.ts` creates the `better-auth` React client pointed at `/api/auth`.
- `components/ui/` are shadcn-generated primitives (config in `components.json`, style `radix-vega`, base color `neutral`); prefer `pnpm dlx shadcn@latest add <component>` over hand-writing new primitives so they stay consistent with the generated ones.
- `components/dashbord/` (note: intentionally/accidentally spelled without the second "a" — match existing spelling when adding files here) holds feed/sidebar/stories/photo-upload feature components.
- Path alias `@/*` maps to the app root (see `tsconfig.json`), matching shadcn's aliases in `components.json`.

## Conventions

- Backend: single quotes, trailing commas (`.prettierrc`); `@typescript-eslint/no-explicit-any` is disabled and `no-floating-promises`/`no-unsafe-argument` are warnings only, not errors.
- New Drizzle tables must live in a file matching `src/**/schema.ts` so `drizzle-kit` (config in `drizzle.config.ts`) picks them up, and must be added to the merged `schema` export in `database/database.module.ts`.
